# Amelia — Integration Reference

How each third-party integration works in this project, what was confirmed during pre-build testing, and what to know before using it in the main build.

*Last updated: March 22, 2026*

---

## 1. Convex — Backend + Database

**Role in Amelia:** All data lives in Convex. Mutations write records (patients, bills, claim batches). Queries read them. HTTP Actions receive Interswitch webhooks. Real-time subscriptions power the dashboard — when a bill is paid, every open browser tab updates instantly without polling.

**How it works:**
Convex is a full-stack backend that runs on their cloud. Your frontend connects to it via a React client (`ConvexProvider`). Functions are written in TypeScript and deployed via `npx convex dev` (development) or `npx convex deploy` (production). There is no REST API to call — queries and mutations are called like typed functions from the frontend.

Two processes must run simultaneously during local development:
- `npx convex dev` — syncs your Convex functions and keeps the backend live
- `bun dev` — runs the Vite frontend

Env vars (`CONVEX_DEPLOYMENT`, `VITE_CONVEX_URL`, `VITE_CONVEX_SITE_URL`) are auto-generated by `npx convex dev` and written directly to `.env.local`. They are already set.

**In the main project:**
- Schema defined in `convex/schema.ts`
- Queries/mutations in `convex/*.ts` files, imported via `api` object
- HTTP Actions in `convex/http.ts` — exposed at `https://majestic-vole-961.convex.site/`
- Convex env vars (API keys for server-side services) set via `npx convex env set KEY value`

## 2. Clerk — Authentication

**Role in Amelia:** Staff login. Two roles: `staff` (front desk, can create bills and patients) and `admin` (can access dashboard, generate claims, see financials). Clerk handles the login UI, session management, and JWT issuance. Convex validates those JWTs server-side on every protected query/mutation.

**How it works:**
Clerk wraps the React app in `ClerkProvider`. The Convex integration uses `ConvexProviderWithClerk` which automatically attaches the Clerk JWT to every Convex request. On the Convex side, `ctx.auth.getUserIdentity()` reads the verified identity — no manual JWT parsing needed.

The connection between Clerk and Convex requires a JWT template in the Clerk dashboard (named exactly `convex`) and a `convex/auth.config.ts` file pointing at your Clerk issuer domain.

Role metadata flows through `identity.publicMetadata.role` — set it in the Clerk dashboard per user. Convex queries can read it to gate access.

**Env vars:**
- `VITE_CLERK_PUBLISHABLE_KEY` — in `.env.local` (frontend)
- `CLERK_JWT_ISSUER_DOMAIN` — in Convex env, not `.env.local` (set via `npx convex env set`)

## 3. Mistral OCR — Document Extraction ✅ Confirmed

**Role in Amelia:** Upload a scanned HMO card or pre-authorisation letter → extract structured data (HMO name, member ID, auth code, coverage limit) → pre-fill the patient registration form. Eliminates manual re-keying of card details.

**How it works:**
The SDK (`@mistralai/mistralai`) exposes `client.ocr.process()`. You pass a document URL or base64-encoded file and the model `mistral-ocr-latest`. It returns a `pages` array — each page has a `markdown` field with the extracted text in clean Markdown format. Tables, headings, and structured data are preserved in the output.

```typescript
const client = new Mistral({ apiKey: process.env.MISTRAL_API_KEY });
const result = await client.ocr.process({
  model: "mistral-ocr-latest",
  document: { type: "document_url", documentUrl: "https://..." }
});
const text = result.pages[0].markdown;
```

**Confirmed:** Tested against an arXiv PDF. Returned clean markdown with headings and body text extracted accurately. Response time ~3–5 seconds for a standard document.

**In the main project:** Called from a Convex action (server-side) since the API key must not be exposed to the browser. The action receives either a Convex file storage URL or an uploaded document, calls OCR, and returns structured JSON parsed by Claude from the markdown output.

**Env var:** `MISTRAL_API_KEY` — set in Convex env (not `.env.local`)

## 4. Inngest — Background Job Workflows ✅ Confirmed

**Role in Amelia:** Async tasks that should not block the user's action. When a bill is created, Inngest sends the SMS payment link. When auth is pending for 6 hours, Inngest reminds staff. When a TPA payment is 14 days overdue, Inngest flags the dashboard. These are durable — if they fail mid-run, Inngest retries automatically.

**How it works:**
You define functions that listen for named events. When `inngest.send({ name: "bill/created", data: { ... } })` is called from a Convex mutation, Inngest receives the event and executes all matching functions asynchronously. Functions can use `step.sleep()` for time-delayed steps (e.g. wait 6 hours, then check if auth code was entered).

The Inngest server needs an HTTP endpoint to call your functions. In development, the Inngest CLI (`inngest-cli dev`) polls your local server. In production, Inngest Cloud calls your deployed endpoint.

**Confirmed setup:**
- Package: `inngest@4.0.2`
- Client: `new Inngest({ id: "clearpath" })`
- Server: Hono app with `serve` from `inngest/hono`, mounted at `/api/inngest`
- Bun-native export: `export default { port: 3000, fetch: app.fetch.bind(app) }` — **do not use `@hono/node-server`**, it causes `internal_server_error` from the Inngest dev server
- App synced as "clearpath", framework "hono", SDK v4.0.2, 1 function found
- Test event `test/ping` received and logged by CLI

**In the main project:**
The Inngest endpoint moves from the test Hono server into `convex/http.ts` as an HTTP Action, served at `https://majestic-vole-961.convex.site/api/inngest`. Events are sent from inside Convex mutations/actions, not from the React frontend. Production requires `INNGEST_EVENT_KEY` and `INNGEST_SIGNING_KEY` in Convex env.

**Env vars:**
- `INNGEST_DEV=1` — in `.env.local` for local mode (skips key validation)
- `INNGEST_EVENT_KEY` + `INNGEST_SIGNING_KEY` — Convex env, production only

## 5. E2B Code Interpreter — PDF Generation Sandbox ✅ Confirmed

**Role in Amelia:** Generates the HMO claims form PDF and Medical Director cover letter PDF. A Python script runs inside an isolated E2B sandbox, receives the bill data and HMO template config, uses `pdf-lib` or `reportlab` to render the PDF with precise layout, and returns the file. Using a sandbox keeps untrusted PDF generation code away from the main server.

**How it works:**
`Sandbox.create()` spins up a cloud microVM with a Python environment in ~3 seconds. You run arbitrary code inside it via `sbx.runCode()`. Files written inside the sandbox can be read back with `sbx.files.read()`. `sbx.kill()` cleans it up. The sandbox is ephemeral — each claim generation gets a fresh one.

```typescript
import { Sandbox } from "@e2b/code-interpreter";
const sbx = await Sandbox.create();
const exec = await sbx.runCode(`
  with open("/tmp/output.txt", "w") as f:
      f.write("Hello from sandbox")
  print("DONE")
`);
const content = await sbx.files.read("/tmp/output.txt");
await sbx.kill();
```

**Confirmed:** Sandbox created, Python code executed, file written and read back correctly, sandbox killed cleanly. Round-trip ~4 seconds.

**In the main project:** A Convex action prepares the bill data and HMO template config as a Python dict, sends it to the sandbox, runs the PDF generation script, reads back the PDF bytes, uploads to Convex file storage, and returns the download URL. The sandbox never needs network access for this — all data is passed in code.

**Env var:** `E2B_API_KEY` — set in Convex env

## 6. Firecrawl — Web Scraping + Agent ✅ Confirmed

**Role in Amelia:** Pre-build pipeline to populate the HMO and TPA directory. Scrape the NHIA accredited facility list and top HMO websites to build a canonical list of HMO names, TPA contacts, and submission addresses. This data loads into the `hmo_templates` table and powers the HMO dropdown in the bill builder.

**How it works:**
Two relevant products:

**Scrape** — single-page extraction. `app.scrape(url, { formats: ["markdown"] })` returns clean markdown of a page's content. Throws on failure (v4 behaviour) — wrap in try/catch. Works well on simple HTML pages like NHIA/government sites. JavaScript-heavy pages (SPAs) return only the nav shell.

**Agent** — autonomous multi-page research. `app.startAgent({ prompt, schema, model, maxCredits })` kicks off an agent that browses multiple pages to answer a structured question. Returns a Zod-schema-validated JSON object. Takes 2–4 minutes — use `getAgentStatus()` polling, not the blocking `agent()` call.

```typescript
// Scrape
const result = await app.scrape("https://nhia.gov.ng", { formats: ["markdown"] });

// Agent (polling pattern)
const { id } = await app.startAgent({
  prompt: "Find all accredited TPAs from the NHIA website",
  schema: z.object({ tpas: z.array(z.object({ name: z.string(), email: z.string() })) }) as any,
  model: "spark-1-mini",
  maxCredits: 2000,
});
let status;
do {
  await new Promise(r => setTimeout(r, 5000));
  status = await app.getAgentStatus(id);
} while (status.status === "running");
```

**Confirmed:**
- Scrape: working on `nhia.gov.ng`, returns clean markdown
- Agent: working, returned structured JSON matching Zod schema after ~3 minutes of autonomous browsing
- `maxCredits: 200` refused immediately — use 2000+
- Zod schema requires `as any` cast due to SDK type mismatch with Zod v4

**Package:** `@mendable/firecrawl-js@4.16.0`
**Env var:** `FIRECRAWL_API_KEY` — set in both `.env.local` and Convex env

## 7. Africa's Talking — SMS ✅ Confirmed

**Role in Amelia:** Two SMS touchpoints: (1) when a bill is created, send the patient their payment link and virtual account details; (2) when payment is confirmed, send a receipt SMS. Triggered by Inngest functions, not synchronously in the request path.

**Note on provider selection:** Termii signup was unavailable during testing. BulkSMSNigeria requires a ₦5,000 minimum deposit before issuing an API key. Africa's Talking was selected instead — free sandbox with a simulator, official Node.js SDK, and reliable Nigerian carrier routing.

**How it works:**
The SDK initialises with a username and API key. Username `"sandbox"` routes all traffic to the AT sandbox — messages appear in the AT simulator at `simulator.africastalking.com` and never reach real phones. Switching to production is just changing the username to your live AT account name and swapping the API key.

```typescript
import AfricasTalking from "africastalking";
const at = AfricasTalking({ username: process.env.AT_USERNAME, apiKey: process.env.AT_API_KEY });
const sms = at.SMS;
await sms.send({
  to: ["+2349067748876"],   // international format required
  message: "Your bill is ₦12,500. Pay to: GTBank 0123456789"
  // no `from` field in sandbox — empty string fails SDK validation
});
```

**Confirmed:** SMS delivered to AT simulator. Response contained `statusCode: 101` and `status: "Success"` with a valid `messageId`.

**Critical gotchas:**
- `from` field must be **omitted entirely** in sandbox — passing `from: ""` throws a Joi validation error from inside the SDK
- `to` must be international format with `+` prefix — `08012345678` is rejected by google-libphonenumber inside the SDK
- In production, `from` becomes your registered sender ID (e.g. `"Amelia"`)

**Package:** `africastalking@0.7.9`
**Env vars:** `AT_USERNAME` + `AT_API_KEY` — Convex env for production; `.env.local` for local testing

## 8. Interswitch — Payments + Identity Verification

**Tested: March 22, 2026**

Interswitch has two completely separate API systems with different credentials, different authentication mechanisms, and different purposes. They cannot be mixed.

| System | Credentials | Auth method | Purpose |
|---|---|---|---|
| **Quickteller Business** | `INTERSWITCH_CLIENT_ID` + `INTERSWITCH_SECRET_KEY` + `INTERSWITCH_MAC_KEY` | SHA-512 hash signing — no OAuth | Web checkout / collecting payments |
| **Developer Marketplace** | `ISW_MARKETPLACE_CLIENT_ID` + `ISW_MARKETPLACE_CLIENT_SECRET` | OAuth 2.0 Bearer token | Identity verification (NIN, bank accounts) |
| **Payment Gateway** (Virtual Account) | Separate OAuth credentials — not yet provisioned | OAuth 2.0 Bearer token, resource: `isw-paymentgateway` | Virtual account creation — requires separate Interswitch onboarding |

---

### Payment System (Quickteller Business)

**There is no OAuth step for payments.** The Quickteller Business credentials cannot obtain an OAuth token — attempts return `400 Bad credentials`. This is by design. Authentication for the payment system is done entirely by SHA-512 hash-signing the payment parameters using the MAC key.

#### Web Checkout ✅ Confirmed

The hosted payment page flow:
1. Build payment parameters: `txn_ref`, `merchant_code`, `product_id`, `pay_item_id`, `amount`, `site_redirect_url`, `currency`, `isswitch`
2. Compute `SHA512(txnref + product_id + pay_item_id + amount + site_redirect_url + mac_key)` — field concatenation order is critical, no separators
3. POST these params to `https://newwebpay.qa.interswitchng.com/collections/w/pay`
4. Interswitch shows a hosted card payment page — your server never touches card data
5. After payment, Interswitch redirects to your `site_redirect_url` with `txnref`, `payRef`, and `ResponseCode` in the query string

`ResponseCode: "00"` means approved. For the hackathon, use this redirect value to confirm payment. In production, add server-side verification via the transaction query endpoint (requires OAuth credentials which Interswitch provisions separately for production).

**Sandbox test card:**
- Number: `5061050254756707864` | Expiry: `06/26` | CVV: `111` | PIN: `1111`

**Critical constraint:** `txnref` must be ≤ 15 characters. Longer values return `"Incorrect Transaction"` on the payment page. Recommended format: `CP` + last 13 digits of `Date.now()`.

**Test file:** `tests/interswitch/payment/02-web-checkout.ts`

#### Virtual Account API — ❌ Not Accessible with Current Credentials

**Endpoint confirmed:** `POST https://qa.interswitchng.com/paymentgateway/api/v1/payable/virtualaccount`

The endpoint exists and responds (not a 404). The blocker is credentials. This API belongs to a **third Interswitch system** — the Payment Gateway OAuth platform — separate from both the Marketplace and the Quickteller web checkout:

- Marketplace OAuth token → `403 "token does not contain resource id (project-x-merchant)"`
- Quickteller Business credentials → `401 Bad credentials` (cannot produce OAuth tokens at all)
- Payment Gateway OAuth project → not available in either developer portal (developer.interswitchgroup.com only shows Marketplace projects)

**What this means:** Virtual accounts require a separate credential set provisioned directly by Interswitch — likely through a business onboarding process. This cannot be self-served from the current dashboard.

**Hackathon decision:** Build with the **web checkout** (already confirmed working) as the payment collection method. The UX difference — patient clicks a link vs. receives a bank account number — is cosmetic for the demo. The underlying webhook reconciliation and bill status update flow is identical. Contact Interswitch post-hackathon to provision Payment Gateway OAuth credentials for virtual accounts in production.

#### Transaction Query ⚠️ Not Working in Sandbox

Requires an OAuth token that Quickteller Business credentials cannot produce in the sandbox environment. For the hackathon: trust the `ResponseCode` in the redirect callback. For production: Interswitch provisions credentials that support both hash-signing and OAuth.

**Test file:** `tests/interswitch/payment/03-transaction-query.ts`

#### Webhook — Not Yet Tested ⚠️

`TRANSACTION.COMPLETED` fires to your Convex HTTP Action when a payment lands. This is what triggers real-time bill status updates. Cannot be tested locally without a public URL (ngrok or a Vercel deploy). Must be tested before Day 2 of the build sprint.

#### OPay Wallet Payment ✅ Confirmed

A second payment channel alongside the web checkout — no OAuth required.

**Initialize:**
```
POST https://qa.interswitchng.com/collections/api/v1/opay/initialize
Content-Type: application/json   (no Authorization header)

{ "merchantCode": "MX180207", "payableCode": "Default_Payable_MX180207", "amount": 100000, "transactionReference": "CP1774205435235" }
```
Returns `{ responseCode: "09", redirectUrl: "https://sandboxcashier.opaycheckout.com/...", authenticationType: "REDIRECT" }`. `responseCode: "09"` = session initialized.

Redirect the patient to the `redirectUrl`. They complete payment in OPay's hosted cashier UI.

Sandbox testing did not give Amelia a reliable callback/return redirect. The safe implementation is: open OPay in a new tab, keep Amelia on the current page, then confirm status from Amelia using the saved `transactionReference`.

**Confirm payment:**
```
POST https://qa.interswitchng.com/collections/api/v1/opay/status
Content-Type: application/json

{ "reference": "CP1774205435235" }
```
Returns `{ responseCode: "00", transactionReference: "CP1774205435235" }`. `responseCode: "00"` = paid and confirmed.

**No webhooks.** OPay uses polling only — call the status endpoint from Amelia after the patient completes wallet checkout, then mark the bill as PAID.

**Sandbox test credentials:**
- Phone: `1259257649` | PIN: `123456` | OTP: `315632` → success | OTP: `315633` → fail

**Amelia use:** Second payment option on the bill payment screen alongside card. Patient chooses "Pay with Card" (web checkout) or "Pay with OPay Wallet". Both methods use the same `transactionReference` tied to the bill.

**Test file:** `tests/interswitch/payment/05-opay.ts`

### Developer Marketplace APIs

All marketplace calls follow the same auth pattern: get a Bearer token once, use it for all subsequent calls.

```typescript
const credentials = Buffer.from(`${clientId}:${clientSecret}`).toString("base64");
const { access_token } = await fetch("https://qa.interswitchng.com/passport/oauth/token", {
  method: "POST",
  headers: { Authorization: `Basic ${credentials}`, "Content-Type": "application/x-www-form-urlencoded" },
  body: "grant_type=client_credentials",
}).then(r => r.json());
```

The JWT payload contains `api-routing-actions` — an array of activated APIs for your project. Currently: `["VerifyMeNin", "UVBankVerification"]`.

All API calls route through `https://api-marketplace-routing.k8.isw.la/marketplace-routing/`.

#### NIN Verification ✅ Confirmed

```
POST https://api-marketplace-routing.k8.isw.la/marketplace-routing/api/v1/verify/identity/nin
Body: { "firstName": "...", "lastName": "...", "nin": "..." }
```

Returns `nin_check.status: "EXACT_MATCH"` when the name matches the NIN record in the NIMC database. Also returns gender, masked phone number, and a base64-encoded photo. If name does not match, returns `"NOT_MATCH"` — not an error, still a `200`.

**Amelia use:** Verify patient identity at registration before submitting an NHIA claim. The NHIA rejects any claim with a missing or unverified NIN.

**Test file:** `tests/interswitch/marketplace/02-nin-verification.ts`

#### Bank Account Verification ✅ Confirmed

Two-step flow:

**Step 1 — Get bank list:**
```
GET https://api-marketplace-routing.k8.isw.la/marketplace-routing/api/v1/verify/identity/account-number/bank-list
```
Returns 100+ Nigerian banks with their `code` field (e.g. `"058"` for GTBank). Use this to populate the bank dropdown during clinic onboarding.

**Step 2 — Resolve account:**
```
POST https://api-marketplace-routing.k8.isw.la/marketplace-routing/api/v1/verify/identity/account-number/resolve
Body: { "accountNumber": "1000000000", "bankCode": "058" }
```
Returns `bankDetails.accountName` — the account holder's registered name. Show it back to the user to confirm before saving.

**Amelia use:** During clinic onboarding, verify the clinic's payout bank account. Prevents failed disbursements from typos.

**Test file:** `tests/interswitch/marketplace/04-bank-account-verification.ts`

#### Transaction Search ❌ Not Activated

Not in `api-routing-actions` on this project's JWT. The endpoint exists but returns an auth error. Not needed for the hackathon — skip.
